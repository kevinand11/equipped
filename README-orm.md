# ORM

A database-agnostic ORM inspired by Ecto, built on top of [valleyed](https://github.com/stranerd/valleyed) validation pipes.

## Architecture

```
schema/     Define data shapes with valleyed pipes + associations
query/      Functional, composable query builder producing a QueryAST
adapters/   Database-specific implementations (MongoDB, PostgreSQL)
repo/       Per-schema repository with CRUD, validation, associations, transactions
```

## Schema

Define schemas using `schema()` with valleyed pipes for each field. Chain `.pk()` to set the primary key, `.computed()` for derived fields, and `.belongsTo()`, `.hasOne()`, `.hasMany()`, `.manyToMany()` for associations.

```ts
import { v } from 'valleyed'
import { schema } from './orm'

const UserSchema = schema({
  id: v.string(),
  name: v.string(),
  email: v.string(),
  age: v.number().pipe(v.gte(0)),
}).pk('id')
  .computed('displayName', v.string(), (data) => `${data.name} (${data.age})`)

const PostSchema = schema({
  id: v.string(),
  title: v.string(),
  body: v.string(),
  userId: v.string(),
}).pk('id')
  .belongsTo('user', () => UserSchema, 'userId')
  .hasMany('comments', () => CommentSchema, 'postId')
```

### Primary Key

The primary key is a regular field in the schema with its own validation pipe. Call `.pk()` to designate it. The pipe's output type determines the type used in `findById`, `updateById`, and `deleteById`.

```ts
// String PK (default for most use cases)
const UserSchema = schema({
  id: v.string(),
  name: v.string(),
}).pk('id')

// Numeric PK
const LegacySchema = schema({
  id: v.number(),
  label: v.string(),
}).pk('id')

// MongoDB convention
const MongoSchema = schema({
  _id: v.string(),
  name: v.string(),
}).pk('_id')
```

The primary key field is excluded from validation on insert (since it's generated by the adapter) and from partial validation on update.

### Computed Fields

Computed fields are derived from other fields at insert time. They have a validation pipe and a compute function that receives the raw data and returns a value validated against that pipe. Chain `.computed(name, pipe, fn)` after the schema to add them — the compute function's `data` parameter is automatically typed to the schema's field outputs.

```ts
const UserSchema = schema({
  id: v.string(),
  firstName: v.string(),
  lastName: v.string(),
}).pk('id')
  .computed('fullName', v.string(), (data) => `${data.firstName} ${data.lastName}`)
  .computed('searchKey', v.string(), (data) => `${data.firstName} ${data.lastName}`.toLowerCase())
```

- On insert (`validateSchema`), computed values are automatically derived from the input data
- On update (`validatePartialSchema`), computed fields can be provided explicitly and are validated against their pipe
- Computed field outputs are included in `InferEntity` — they appear on the entity type just like regular fields

### Associations

Four association types are supported, all using lazy refs `() => Schema` to avoid circular dependencies. Each is a method on the schema builder:

| Method | Meaning | Foreign Key Location |
|--------|---------|---------------------|
| `.belongsTo(name, () => Target, 'fk')` | This schema has the FK | On this schema |
| `.hasOne(name, () => Target, 'fk')` | Target has the FK pointing here | On target schema |
| `.hasMany(name, () => Target, 'fk')` | Target has the FK pointing here | On target schema |
| `.manyToMany(name, () => Target, () => Join, 'thisFk', 'thatFk')` | Through a join table | On join schema |

### Validation

Two helpers validate data against a schema's pipes:

- `validateSchema(schema, data)` — validates all fields except the primary key (for inserts), runs compute functions for computed fields, throws on invalid/missing
- `validatePartialSchema(schema, data)` — validates only provided fields (for updates), skips the primary key, validates computed field values if present, ignores unknown keys

### Type Inference

| Type | Purpose |
|------|---------|
| `InferEntity<F, C>` | Output types from field + computed pipes |
| `InferInput<F, PK>` | Input types for field pipes, excluding the primary key |
| `SchemaEntity<S>` | Extract entity type from a schema |
| `SchemaInput<S>` | Extract input type from a schema |
| `SchemaPrimaryKeyType<S>` | Extract the primary key's output type from a schema |
| `WithPreloaded<E, S, K>` | Entity with a preloaded association key |
| `SelectFields<E, S>` | Pick specific fields from an entity |
| `SchemaRawShape<D>` | Map field defs to their output types (used for computed data typing) |
| `FieldsOf<D>` | Extract regular field defs from a mixed defs record |
| `ComputedsOf<D>` | Extract computed field defs from a mixed defs record |
| `AssociationsOf<D>` | Extract association defs from a mixed defs record |

## Query Builder

Functional, composable query construction. All helpers return plain objects that get compiled into a `QueryAST`.

```ts
import { query, where, and, or, orderBy, limit, offset, select, raw, eq, gt, isIn } from './orm'

const ast = query(
  where('status', eq('active')),
  and(where('age', gte(18)), where('age', lte(65))),
  or(where('role', eq('admin')), where('role', eq('mod'))),
  orderBy('name'),
  orderBy('createdAt', 'desc'),
  limit(20),
  offset(40),
  select('name', 'email', 'role'),
)
```

### Condition Helpers

| Helper | Condition |
|--------|-----------|
| `eq(value)` | Equal |
| `ne(value)` | Not equal |
| `gt(value)` | Greater than |
| `gte(value)` | Greater than or equal |
| `lt(value)` | Less than |
| `lte(value)` | Less than or equal |
| `isIn(values)` | In set |
| `notIn(values)` | Not in set |
| `like(pattern)` | Pattern match (ILIKE in PG, $regex in Mongo) |
| `exists()` | Field exists / is not null |
| `notExists()` | Field does not exist / is null |

### Query Operators

| Operator | Purpose |
|----------|---------|
| `where(field, condition)` | Filter by field |
| `and(...clauses)` | Combine with AND |
| `or(...clauses)` | Combine with OR |
| `orderBy(field, 'asc'\|'desc')` | Sort (default: asc) |
| `limit(n)` | Limit results |
| `offset(n)` | Skip results |
| `select(...fields)` | Project specific fields |
| `raw(value)` | Pass adapter-specific data (escape hatch) |
| `query(...ops)` | Compile operators into a QueryAST |

`and` and `or` support arbitrary nesting.

## Adapters

Adapters implement the `Adapter` interface to translate `QueryAST` into database-specific operations.

### Adapter Interface

```ts
interface Adapter {
  connect(): Promise<void>
  disconnect(): Promise<void>
  findMany(schema, table, queryAst): Promise<Record<string, unknown>[]>
  findOne(schema, table, queryAst): Promise<Record<string, unknown> | null>
  insertOne(schema, table, data, options?): Promise<Record<string, unknown>>
  insertMany(schema, table, data[], options?): Promise<Record<string, unknown>[]>
  updateMany(schema, table, queryAst, data, options?): Promise<Record<string, unknown>[]>
  updateOne(schema, table, queryAst, data, options?): Promise<Record<string, unknown> | null>
  upsertOne(schema, table, queryAst, data, options?): Promise<Record<string, unknown>>
  deleteOne(schema, table, queryAst): Promise<Record<string, unknown> | null>
  deleteMany(schema, table, queryAst): Promise<Record<string, unknown>[]>
  count(schema, table, queryAst): Promise<number>
  session<T>(callback: () => Promise<T>): Promise<T>
  paginatedQuery(schema, table, queryAst, pagination): Promise<PaginatedResult>
}
```

### TableConfig

Each adapter defines its own table config type extending `BaseTableConfig`:

```ts
// Base (shared)
type BaseTableConfig = { primaryKey: string }

// MongoDB
type MongoTableConfig = { db: string; col: string; primaryKey: string }

// PostgreSQL
type PgTableConfig = { schema?: string; table: string; primaryKey: string }
```

The `primaryKey` is read from the schema itself (via `.pk()`), not from the table config passed to `repo.for()`.

### Insert/Update/Upsert Options

- `InsertOptions` — `makeId?(index) => unknown`, `getTime?() => Date`
- `UpdateOptions` — `getTime?() => Date`
- `UpsertOptions` — `makeId?() => unknown`, `getTime?() => Date`

`makeId` returns `unknown` to support any primary key type (string, number, etc.).

### MongoDB Adapter

```ts
import { MongoAdapter } from './orm'

const adapter = new MongoAdapter({ uri: 'mongodb://localhost:27017' })
await adapter.connect()
```

- Uses the schema's `primaryKey` field name directly (no automatic `id` ↔ `_id` mapping)
- Generates `ObjectId` strings for IDs by default
- Uses `AsyncLocalStorage` for session/transaction propagation
- Supports raw MongoDB operators via `raw()` (e.g. `$inc`, `$push`)
- Query compiler: `compileMongoQuery(ast, primaryKey)` → `{ filter, sort, limit, skip, projection }`
- Update compiler: `compileMongoUpdate(data, raws, now)` → MongoDB update document with `$set` + raw operators

### PostgreSQL Adapter

```ts
import { PgAdapter } from './orm'

const adapter = new PgAdapter({ pool: pgPool })
```

- Expects a `pg`-compatible Pool (anything with `.query(sql, params)`)
- Generates UUIDs for IDs by default via `crypto.randomUUID()`
- Uses `AsyncLocalStorage` for session/transaction propagation
- Upsert uses `ON CONFLICT ... DO UPDATE`
- Query compiler: `compilePgQuery(ast, tableName, primaryKey)` → parameterized SQL clauses
- SQL builders: `buildSelectQuery`, `buildCountQuery`, `buildInsertQuery`, `buildUpdateQuery`, `buildDeleteQuery`

## Repo

The Repo provides a per-schema, high-level API that ties together validation, the query builder, and an adapter.

```ts
import { repo } from './orm'

// For MongoDB adapter
const UserRepo = repo({ adapter, schema: UserSchema, config: { db: 'mydb', col: 'users' } })

// For PostgreSQL adapter
const UserRepo = repo({ adapter, schema: UserSchema, config: { schema: 'public', table: 'users' } })
// primaryKey is read from the schema's .pk() call
```

### CRUD Operations

```ts
// Find
const user = await UserRepo.findById('abc')
const user = await UserRepo.findOne(where('email', eq('john@example.com')))
const users = await UserRepo.findMany(where('age', gt(18)), orderBy('name'), limit(10))

// Insert (validates all fields against schema)
const created = await UserRepo.insert({ name: 'John', email: 'john@test.com', age: 25 })
const many = await UserRepo.insertMany([data1, data2])

// Update (validates only provided fields)
const updated = await UserRepo.updateById('abc', { name: 'Jane' })
const all = await UserRepo.update([where('status', eq('old'))], { status: 'archived' })

// Delete
const deleted = await UserRepo.deleteById('abc')
const all = await UserRepo.delete(where('status', eq('archived')))

// Upsert (validates insert data fully, update data partially)
const result = await UserRepo.upsert(
  [where('email', eq('john@test.com'))],
  { insert: { name: 'John', email: 'john@test.com', age: 25 }, update: { age: 26 } },
)

// Count
const n = await UserRepo.count(where('status', eq('active')))

// Paginated query
const page = await UserRepo.paginatedQuery(
  [where('status', eq('active')), orderBy('createdAt', 'desc')],
  { page: 1, limit: 20 },
)
// page.results, page.pages.{current,last,next,previous}, page.docs.{total,count,limit}
```

### Implicit Validation

All mutations validate data against the schema before passing to the adapter:

- `insert` / `insertMany` — full validation via `validateSchema` (all fields required)
- `update` / `updateById` — partial validation via `validatePartialSchema` (only provided fields)
- `upsert` — insert data is fully validated, update data is partially validated

Invalid data throws before any database call is made.

### Associations & Preloading

Preload resolves associations via additional queries:

```ts
// Single entity
const post = await PostRepo.findById('p1')
const withUser = await PostRepo.preload(post, 'user')
// withUser.user is the related User entity

// Batch (efficient — single query for all entities)
const posts = await PostRepo.findMany()
const withUsers = await PostRepo.preloadMany(posts, 'user')
```

`preloadMany` batches related lookups into a single query using `isIn`, then maps results back by foreign key. Extra query ops can be passed after the key to filter/sort the associated records.

### Transactions

All adapters use `AsyncLocalStorage` to propagate transaction context. Any repo operation called inside the callback automatically participates in the transaction.

```ts
await UserRepo.session(async () => {
  const user = await UserRepo.insert({ name: 'Alice', email: 'alice@test.com', age: 30 })
  await PostRepo.insert({ title: 'Hello', body: '...', userId: user.id })
  // if either throws, the whole transaction rolls back
})
```

Nested `session()` calls are no-ops — they reuse the existing transaction. Multiple repos sharing the same adapter share the same transaction context.

## Adding a New Adapter

Implement the `Adapter` interface with a query compiler that translates `QueryAST` to your database's query language. See `adapters/mongo/` and `adapters/pg/` as reference implementations.
